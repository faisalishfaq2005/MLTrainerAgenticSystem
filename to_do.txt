2 . architectural design 
Yes, that is the right mental model.

In a multi-user web app, you usually do not want one shared “live conversation state” for everyone. Instead, each user interaction starts a separate job with its own job_id, and all mutable runtime state for that user stays attached to that job. So user A and user B can both be active at the same time, but they each have isolated state, isolated intake progress, isolated collected_info, and eventually isolated training/deployment results.

What should be shared is only the code and services that are safe to reuse, like the orchestrator class definition, the graph definition, prompt templates, and maybe a database connection pool. What should not be shared globally is per-user state like conversation history, selected model provider, API key, current stage, or router credentials.

A simple flow looks like this:

User logs in and clicks “start training”.
Backend creates a new job_id for that user.
That job gets its own intake state and its own intake agent instance.
The user chats with that job until enough info is collected.
The intake step stores collected_info in that job only.
The orchestrator creates the downstream router for that job using that job’s credentials.
The rest of the pipeline runs only for that job.
Results are saved back to that job record.
So yes, your design should be “one job per user request/session,” and the job should own the user-specific runtime objects. But there is one nuance: the orchestrator itself does not have to be a separate Python object per user in the strict sense. It can be one server-level service that manages many jobs, as long as each job has its own isolated state and object instances where needed.

The important rule is this: do not put user-specific credentials or conversation state in module globals or shared singleton state. If you do that, user A’s data can leak into user B’s run.
